# Pochak OTT Platform — CLAUDE.md

## Project Structure

TypeScript 모노레포 (pnpm workspaces) + Java Spring Boot MSA.

```
services/        → 9개 Spring Boot 마이크로서비스 (Java 21)
clients/apps/    → web-front (Vite+React), bo-web (Next.js 16), ios, android
clients/packages → @pochak/api-client, domain-types, design-tokens, utils
db/migrations/   → Flyway SQL 마이그레이션
docs/            → 프로젝트 문서, API 스펙, 정책
```

tsconfig 편집 시 해당 패키지 파일만 staging한다. 상위 디렉토리 변경을 diff에 포함하지 않는다.

## Multi-Agent Workflow

이 프로젝트는 `.claude/agents/`에 정의된 멀티 에이전트 워크플로우를 따른다.

**에이전트 역할**: Planner → Backend Dev / Frontend Dev → Tester → Reviewer → main

**핵심 규칙**:
1. main 직접 작업 금지. `pck/PCK-NNN-설명` 브랜치 사용
2. 역할 경계 준수: Backend는 `services/`만, Frontend는 `clients/`만
3. 한 Task(PCK-NNN) = 한 브랜치 = 한 PR
4. 완료 = 테스트 통과 + 커밋. "된 것 같다"는 완료가 아님
5. Reviewer 승인 없이 main 머지 금지

자세한 워크플로우는 `.claude/agents/WORKFLOW.md` 참조.

## General Rules

- 요청 범위를 초과하는 변경을 하지 않는다. 최소한의 타겟 수정만 수행
- 변경 전 사용자 확인 없이 스코프를 확장하지 않는다
- 30분 이상 같은 에러로 막히면 사람에게 에스컬레이션

## Styling / CSS

Tailwind CSS v4 알려진 이슈:
- `space-y-*`, `space-x-*` → 동작 안 하면 `gap-*` 또는 인라인 `margin` 사용
- `mt-*`, `mb-*` 등 유틸리티 클래스가 적용 안 되면 → 즉시 인라인 스타일 전환
- `tailwind-merge` 충돌 → 인라인 스타일 우선
- **디버깅에 시간 낭비하지 않는다. 안 되면 인라인으로 전환.**

## UI Implementation

- Figma/PDF 디자인 구현 시, 한 번에 한 컴포넌트씩 구현
- 각 컴포넌트 완료 후 시각적 확인을 거친 뒤 다음으로 진행
- 여러 파일에 걸친 대규모 UI 변경 금지
- 디자인 스펙 없이 임의 UI 결정 금지

## Build & TypeScript

변경 후 반드시 빌드 확인:
- 백엔드: `./gradlew test`
- 프론트엔드: `pnpm tsc --noEmit && pnpm test`
- rootDir 충돌, DOM/Node API 불일치, 타입 선언 누락 주의

## Testing

- TDD 원칙: 테스트 먼저, 구현 나중
- 커버리지 90% 이상 목표
- Mock은 외부 의존성에만 사용 (내부 로직은 실제 코드로 테스트)

## Commit Convention

```
feat(PCK-NNN): 한줄 설명
fix(PCK-NNN): 한줄 설명
test(PCK-NNN): 한줄 설명
```
